1. Общие сведения о проекте
1.1 Цель проекта
Разработать веб-сайт туристического агентства «Слетать.ру» и внутреннюю систему для сотрудников, объединяющую приём заявок с сайта, их обработку менеджерами, ведение задач, публикацию статей и коммуникацию с клиентами (email-рассылки, чат на сайте) с автоматической передачей обращений в систему U-ON.
1.2 Заказчик
Сеть туристических агентств «Слетать.ру», г. Пенза.
1.3 Референс дизайна и структуры
В качестве референса для дизайна и структуры публичного сайта используется предоставленный заказчиком макет главной страницы (приложен отдельным файлом). Ключевые элементы референса: светлая цветовая палитра (синий + золотой акцент), крупный герой-блок с фото, блоки преимуществ с иконками, карточки услуг с фотографиями, блок отзывов, пошаговая схема бронирования, контактная информация офиса и карта проезда.
1.4 Общая архитектура решения (кратко)
Система состоит из двух взаимосвязанных частей:
Публичный сайт — витрина услуг, приём заявок, блог со статьями;
Внутренняя панель (админка) — мини-CRM для менеджеров: заявки, канбан-доска задач, дашборд, управление статьями, настройки рассылок.
Данные обеих частей хранятся в единой базе данных на сервере, размещённом на территории РФ (требование 152-ФЗ по хранению персональных данных клиентов). Для обращения к отдельным внешним сервисам, недоступным из РФ, предусматривается точечный прокси/VPN, не затрагивающий основной трафик сайта и CRM.

Единая форма заявки (модальное окно и/или встроенный блок), поля:
Имя (обязательно)
Телефон (обязательно)
Email (опционально)
Направление / тип тура (опционально, выпадающий список)
Комментарий (опционально)

Логика обработки отправленной заявки:
Запись заявки в базу данных сайта (таблица заявок, статус «Новая»).
Автоматическая отправка обращения в U-ON через API.
Email-уведомление ответственному менеджеру / общей почте отдела продаж.
Автоматическое письмо клиенту с подтверждением получения заявки (опционально, обсуждается с заказчиком).
3.3 Требования к дизайну
Адаптивная вёрстка (desktop / tablet / mobile), mobile-first подход к формам;
Цветовая палитра и стиль — по референсу заказчика (синий + золотой акцент, светлый фон, скруглённые карточки);
Оптимизация изображений (WebP, ленивая загрузка) для быстрой загрузки страниц;
Доступность (контрастность текста, читаемые размеры шрифта).
3.4 SEO-требования
Уникальные мета-теги (title, description) для каждой страницы и статьи;
Микроразметка Schema.org (Organization, TravelAgency, BreadcrumbList);
Автогенерация sitemap.xml и настройка robots.txt;
Человекопонятные URL (ЧПУ) для страниц и статей.

6. Канбан-доска задач
Настраиваемые колонки (например: Новая → В работе → Ждём клиента → Готово);
Карточка задачи: заголовок, описание, ответственный, дедлайн, необязательная привязка к заявке из CRM;
Перетаскивание карточек между колонками (drag-and-drop);
Уведомления о приближающихся и просроченных дедлайнах;
Личная доска менеджера + общая доска отдела (для руководителя).

7. Дашборд для менеджеров
Сводная панель с ключевыми показателями за выбранный период:
Количество новых обращений/заявок;
Заявки в работе с разбивкой по статусам воронки;
Конверсия по этапам воронки;
Комиссия — по менеджеру (личная) и по отделу (для руководителя);
График динамики заявок по дням/неделям.

8. Интеграция с U-ON
8.1 Принцип работы
При создании заявки любым способом (форма на сайте, чат-бот, звонок, зафиксированный менеджером вручную) бэкенд системы отправляет запрос к REST API U-ON для создания обращения, используя API-ключ, выданный в личном кабинете U-ON.
8.2 Требования
Получить от заказчика доступ к личному кабинету U-ON и API-ключ;
Реализовать очередь повторных попыток отправки (через Redis) на случай временной недоступности U-ON, чтобы заявки не терялись;
Логировать все запросы к U-ON для диагностики сбоев интеграции;
Сохранять ID обращения U-ON в карточке заявки для сопоставления записей между системами.

9. Интеграция с email и рассылки
9.1 Транзакционные письма
Подтверждение заявки клиенту;
Уведомление менеджеру/отделу о новой заявке;
Отправка через внешний SMTP-сервис (рекомендация — российский провайдер, например Unisender или SendPulse, чтобы транзакционные письма не зависели от прокси/VPN).
9.2 Маркетинговые рассылки клиентам
Сегментация базы подписчиков (по интересам, направлениям, статусу клиента);
Конструктор/шаблоны писем;
Статистика открытий и переходов по ссылкам;
Обязательная кнопка отписки и хранение согласия клиента на получение рассылки (требование 152-ФЗ и антиспам-политик почтовых провайдеров).

10. Чат-бот на сайте
10.1 Рекомендуемое решение
Готовый виджет онлайн-чата (JivoSite или Talk-Me) — оба сервиса имеют встроенную интеграцию с U-ON «из коробки», что позволяет обойтись без разработки собственного бота на первом этапе.
10.2 Требования к настройке
Установка виджета на сайт, брендирование под фирменный стиль;
Настройка передачи диалогов/контактов из чата в U-ON как обращений;
Маршрутизация чатов на дежурного менеджера / уведомления о новых сообщениях.
Опция на будущее: переход на бота с собственной сценарной логикой (квалификация лида вопросами до передачи менеджеру) — рассматривается отдельным этапом после запуска, при необходимости.

11. Инфраструктура и хостинг
11.1 Сервер
Хостинг на территории РФ (например, Timeweb Cloud, Selectel, VK Cloud);
Стартовая конфигурация: 2–4 vCPU, 4–8 ГБ RAM, 50–80 ГБ SSD — с возможностью масштабирования при росте нагрузки.
11.2 Домен и SSL
Регистрация домена у аккредитованного регистратора;
SSL-сертификат (Let's Encrypt, автопродление).
11.3 Точечный прокси / VPN
Полный VPN-туннель на весь сервер не используется — вместо этого разворачивается отдельный прокси-контейнер (например, на базе Squid или WireGuard со split-tunneling), через который проксируются запросы только к конкретным внешним сервисам, недоступным из РФ. Список таких сервисов уточняется по ходу разработки, когда конкретный внешний API или пакет окажется заблокирован.
11.4 Резервное копирование
Ежедневное автоматическое резервное копирование базы данных;
Хранение бэкапов отдельно от основного сервера (другой регион/провайдер).
11.5 Соответствие 152-ФЗ
Персональные данные клиентов (заявки, контакты) хранятся исключительно на сервере, физически расположенном на территории РФ.

13. Нефункциональные требования
13.1 Безопасность
Работа сайта и админки только по HTTPS;
Хэширование паролей пользователей, защита от CSRF/XSS/SQL-инъекций;
Ролевая модель доступа (менеджер / руководитель / администратор);
Журналирование ключевых действий в админке (кто и когда изменил заявку/статус).
13.2 Производительность
Время загрузки публичных страниц — не более 2 секунд при обычной нагрузке;
Одновременная работа 5+ менеджеров в админке без заметной деградации отклика.
13.3 Доступность и отказоустойчивость
Мониторинг доступности сайта и сервера (uptime-мониторинг);
Регулярные резервные копии с проверкой возможности восстановления.

15. Критерии приёмки
Все модули функционируют в соответствии с настоящим ТЗ;
Заявка, отправленная с сайта, одновременно появляется в мини-CRM и в U-ON;
Тестовая email-рассылка успешно доставляется группе получателей;
Чат-бот передаёт диалоги/контакты в U-ON;
Дашборд отображает актуальные данные по заявкам и комиссиям;
Пройдено базовое нагрузочное и security-тестирование;
Персональные данные клиентов хранятся на сервере в РФ.

16. Открытые вопросы (требуют уточнения с подрядчиком/заказчиком)
Точный бюджет и предпочтительные сроки запуска;
Тариф и лимиты API в текущем договоре с U-ON;
Итоговый выбор чат-виджета — JivoSite или Talk-Me (сравнение тарифов и функций);
Необходимость автоответа клиенту сразу после отправки заявки;
Список внешних сервисов, требующих точечного прокси/VPN — уточняется по ходу разработки.
